## G1 — Agent CLI failed to launch in a pane; boot block dumped to bash shell
At launch, 1 of 6 agent panes (pane 3, agent-broker-helper) never started
claude-oss — the pane sat at a bash prompt (`bash-3.2$`, title = hostname),
so the entire injected boot block (prompt text, "DO NOT CONTINUE…",
"### PASTE HANDLING", `/opsx:apply agent-broker-helper`) was typed into the
SHELL line-by-line, every line erroring (`bash: /opsx:apply: No such file or
directory`, `bash: When: command not found`, …). The other 5 panes launched
fine and hit the normal boot-curl prompt. So this is an intermittent
per-pane CLI-launch race/failure, not a config problem. Impact: that agent
does no work + the boot block is wasted on the shell. Recovery required
manual relaunch. Fix candidates: verify the CLI process started in each pane
before sending the boot block (poll for the CLI's prompt / a readiness
marker) and retry/relaunch if the pane is still a bare shell; or send the
boot block only after a per-pane readiness check. → v0.8.0/v0.9.0 launch
robustness. Candidate slug: `launch-readiness-check`.

## G2 — `git paw remove` leaves an orphan pane; add/remove re-tile drops an agent's pane
Recovering G1 via `git paw remove feat/agent-broker-helper --force` + `git paw add
--from-spec agent-broker-helper` produced two failures: (a) remove did NOT kill the
failed agent's tmux pane (it was a bash/zsh shell, not the expected CLI process), so
an orphan `zsh` pane lingered and skewed the grid; (b) the add/remove re-tile dropped
a DIFFERENT agent's pane entirely — `dashboard-drop-summary-column` ended with no
tmux pane while still present in the session JSON + broker roster (now a stale/dead
entry). Net: session JSON ↔ tmux desync with no clean command to re-attach a pane to
an existing session agent. Fix candidates: remove must kill its pane by pane-id
regardless of the running process; add/remove re-tile must preserve all OTHER agents'
panes; reconcile session-JSON agents against live panes. → **v0.8.0** (user: "needs
to be fixed in 0.8.0"). Candidate slug: `add-remove-pane-integrity`.

## Conflict-detector escalation noise — additive overlaps flagged as human-input conflicts
With 5 agents running, the broker conflict detector escalated FOUR `agent.question`
"in-flight conflict … has not resolved within 120s. Human input requested." items
to the supervisor inbox. Supervisor diff-audit of the actual hunks vs the
feat/v0.8.0-specs base showed 3 of the 4 were purely ADDITIVE (no real merge risk):
- `src/config.rs` (dev-allowlist ↔ worktree-embedded): different structs
  (`CommonDevAllowlistConfig` vs `LayoutConfig`/`PawConfig`); the only shared fn,
  `generate_default_config`, has hunks ~75 lines apart → clean merge.
- `src/main.rs` (dev-allowlist ↔ worktree-embedded): shared `cmd_supervisor` edits
  are 112 lines apart; everything else is in distinct fns → clean merge.
- `assets/agent-skills/supervisor.md` (coding-agent ↔ dev-allowlist): hunks at L291
  vs L454 → clean merge.
Only `assets/agent-skills/coordination.md` (coding-agent ↔ dev-allowlist) is a TRUE
overlap — both insert at anchor L82 (~2-line doc-insertion collision, trivially
resolved by serializing the merge + rebasing the second branch).
Finding: the detector flags any same-file in-flight overlap and, after a 120s
window, escalates each to the human as an `agent.question` — but it has no notion of
*additive vs conflicting* hunks, so 3/4 escalations here were false alarms that, in
an unattended run, would have stalled work waiting on human input that wasn't needed.
Candidate refinement (v0.9.0): before escalating, have the detector (or supervisor
tooling) test whether the overlapping hunks are line-adjacent / same-anchor vs
disjoint, and only escalate true collisions; downgrade additive overlaps to an
informational `agent.feedback` ("shared file, additive — resolve at merge"). Slug:
`conflict-detector-additive-vs-true`. Supervisor handling this cycle: NOT escalated
to user (no genuine ambiguity); let all agents race; merge plan serializes the
coding-agent ↔ dev-allowlist pair so the second rebases the coordination.md L82
collision.

## F8 CORRECTION (2026-06-24) — it's concurrent-test port collision, NOT live-session collision
Earlier I framed the E2E verify failures (cli_supervisor_no_config, e2e_supervisor_stop
exit 101) as "the tests collide with the live dogfood broker:9119 + tmux." That is WRONG.
Verified in code: the tests isolate tmux via TMUX_TMPDIR (private socket) AND pick a broker
port via `pick_broker_port() = 24_000 + (std::process::id() % 200)` (tests/e2e_supervisor_stop.rs).
So: (a) the live broker is on 9119, OUTSIDE the 24000-24199 test range — no live-session
collision; (b) worktree file-isolation is fine. The REAL cause: the port scheme gives only
200 ports keyed on PID mod 200, so when N agents run `cargo test` CONCURRENTLY (5 at once in
this dogfood, plus the supervisor's verify worktree), two test processes' PIDs collide mod 200
→ same port → "address already in use" → the port-free assertion fails → exit 101. It's a
concurrent-self-collision + general load (N simultaneous brokers/tmux servers), matching
[[feedback_serialize_e2e_verification_runs]]. FIX (sharpens F8): replace pick_broker_port's
PID-mod-200 scheme with an OS-assigned ephemeral port (bind 127.0.0.1:0, read back the
assigned port) so it's collision-proof at any concurrency; and/or serialize verification runs.
Corollary: my "run --lib only, the full suite collides with the live session" nudge to agents
had the right action (skip the flaky E2E, faster) but the WRONG reason — the collision is
concurrency among the agents' own test runs, not the live session.

## G4 — Supervisor falls back to raw curl for rich agent.status (sweep.sh helper too narrow)
Observed: the supervisor publishes a structured agent.status via raw
`curl -X POST .../publish -d '{...,"phase":"audit","detail":{"branch":...,"audit_step":"spec"}}'`
even though sweep.sh exists and it uses `sweep.sh status-publish` for SIMPLE status. Cause:
sweep.sh's status-publish only accepts a plain message string — it doesn't cover the full
agent.status payload (phase + detail.{branch,audit_step,...}), so the supervisor hand-rolls
raw curl for richer status. Same anti-pattern as F1 (agents' raw boot-curl) but supervisor-side:
the bundled helper's surface is narrower than the messages the skill actually needs, so raw
curl leaks back in — verbose, error-prone, and requiring a broad curl allowlist. Fix: extend
the bundled broker helper (sweep.sh, or a unified broker.sh per F1) to cover the FULL
agent.status shape (phase, detail) and the other message types the supervisor/agents emit;
update the supervisor + coordination skills to ALWAYS route broker calls through the helper
(never raw curl). Pairs with F1 (agent-broker-helper) → a single bundled broker-interaction
helper for both roles, allowlisted by path. → v0.8.0/v0.9.0 approval+helper work.
