# v0.7.0 dogfood findings — 2026-06-19

Running log of findings from the v0.7.0 MCP-release dogfood session
(supervisor + 3 coding agents: mcp-server, broker-live-watch-registration,
learnings-privacy-disclosure). Triaged into MILESTONE / changes at cycle end.

## F1 — Agent boot-publish should be a bundled broker helper, not raw curl
All 3 coding agents dead-stalled ~37 min on the boot-block `curl …/publish`
approval prompt; auto-approve never cleared them; supervisor was slow. Root
cause: boot block runs raw `curl` (4×) and the "fix" seeds a broad `curl *`
allowlist grant. Proposed: a bundled `assets/scripts/broker.sh` (agent-side
analogue of `sweep.sh`) wrapping ALL agent→broker curls, installed by
`git paw init`, called from the boot block, allowlisted by precise script
path (least privilege). Keep it a script, NOT a `git paw` subcommand (a
human might run a subcommand and hit errors). → **v0.8.0** approval overhaul.
Candidate slug: `agent-broker-helper`. (Also in memory.)

## F2 — Auto-approve subsystem not clearing dev-essential stalls
Throughout the session every agent repeatedly stalls on safe dev prompts
(file edits, `cargo check`, `cargo test`, dir-access, crate research) that
the `[supervisor.auto_approve]` thread + supervisor sweep should batch-clear.
The human babysitter is clearing them each cycle instead. Direct evidence for
drift F (broker-as-trigger / supervisor-as-actor, least-privilege approvals).
→ **v0.8.0**.
- Refinement: the CLI's "don't ask again for: <cmd>" appears to whitelist by
  **exact string** for some commands, so agents' `echo "... exit $?"` wrappers
  (varying text each time) re-prompt endlessly even after approval, while
  prefix-matchable ones (`python3 -c`, `cargo check *`) generalize fine. Two
  angles: (a) prefer prefix grants; (b) agents shouldn't wrap dev commands in
  `echo "...$?"` exit-code probes that defeat whitelisting.

## F3 — Dashboard broker-log panel could be taller
The broker message-log panel (the `dashboard-broker-log` capability, v0.6.0)
gets too little vertical space relative to the agent table — give it a taller
share so more broker messages are visible at once. Small layout tweak to the
dashboard row-height proportions. → candidate: small dashboard-polish change
(v0.7.x or fold into v0.8.0 dashboard work). Candidate slug: `dashboard-broker-log-height`.

## F5 — Approval keystrokes land as stray chat input when no prompt is active
The learnings-privacy agent reported: "I treated them as stray keystrokes
(likely sweep/approval input landing in the pane) and proceeded." When an
approver (sweep.sh, auto-approve, or a human babysitter) sends `1`/`2`+Enter
to a pane whose prompt has already cleared (agent moved on, or another
approver won the race), the digits land as a literal user message in the
agent's chat input — polluting its context and potentially leaving stray
commands (e.g. an unsubmitted `/opsx:archive`) in the input box. This is the
blind-send-keys race that drift F (broker-as-trigger / supervisor-as-actor)
is meant to eliminate. Mitigations: (a) re-confirm an active "Do you want to
proceed?" prompt immediately before sending approval keys; (b) prefer a
broker-mediated approval signal over raw send-keys. → reinforces **v0.8.0**
drift F. Also a babysitting-procedure note: verify prompt presence per-pane
right before each send; never send Enter to a pane without a live prompt.

## F10 — `assume-unchanged` bit on AGENTS.md silently blocked a required commit
mcp-server needed to add the `rmcp` row to AGENTS.md's approved-dependency
table (required by the proposal's Impact section). The edit was made on disk
but `git status` reported AGENTS.md UNMODIFIED and it couldn't be staged/committed
— because AGENTS.md had the **`assume-unchanged`** index bit set
(`git update-index --assume-unchanged AGENTS.md`). This cost ~5 babysit cycles
of confusion (on-disk had the row, committed didn't, status clean, not gitignored)
until mcp-server diagnosed it and ran `git update-index --no-assume-unchanged`.
Likely a git-paw worktree-setup artifact (assume-unchanged set on AGENTS.md —
perhaps to stop agents mutating the shared conventions doc, or a side effect of
the CLAUDE.md→AGENTS.md symlink handling). Problem: it silently blocks LEGITIMATE
required edits (approved-set updates) and is near-undebuggable without knowing
to check the index bit. Fix options: don't set assume-unchanged on AGENTS.md in
worktree setup; or document/surface it; or have the boot block warn agents.
→ investigate git-paw's worktree/init setup; **v0.8.0** (or sooner if it's a
setup bug). Candidate slug: `agents-md-assume-unchanged`.

## F9 — Supervisor verifies a stale first-commit SHA, not the branch tip
mcp-server made 3 commits (8c63252 impl → 48c9625 E2E tests → 14e5f54 docs).
The supervisor set up its verify worktree at the FIRST commit (8c63252) and ran
ALL five gates against that stale snapshot without advancing to the tip
(14e5f54) as the agent added the tests + docs commits. Result: at the doc-audit
gate it flagged docs (mcp.md, README MCP section) as MISSING and was about to
send mcp-server a doc fix-request — but those docs ARE committed at the tip
(commit 14e5f54), just absent from the verified first-commit snapshot. A
change-level gate (docs/tests present?) must verify the full change at the
branch TIP, not one commit. Relates to [[feedback_verify_against_merge_base_not_tip]].
Fix: supervisor re-resolves and re-checks out the branch tip before
change-level gates (or re-verifies on each new commit against the latest tip).
→ **v0.8.0** (verification correctness).
NOTE: one part of its finding is REAL — the rmcp AGENTS.md approved-set entry
is uncommitted (present on disk in the worktree but in none of the 3 commits),
so mcp-server genuinely needs to `git add AGENTS.md && commit` that approved-set
edit. So the correct disposition: re-verify docs against the tip (they pass) +
have mcp-server commit the AGENTS.md rmcp entry (small real gap).

## F8 — In-session verification false-negatives: E2E tests collide with the live dogfood
Supervisor gate-1 (`just verify`) on learnings-privacy (29de070) exited 101
with ALL 1490+ unit tests green; the only failures were 2 targets —
`cli_supervisor_no_config` and `e2e_supervisor_stop` — both supervisor/broker
**E2E** tests. They spin up their own tmux + broker and collide with the LIVE
dogfood session (broker already bound :9119, `paw-git-paw` tmux session
present). The isolated verify worktree (`.git-paw/tmp/`) isolates files/branch
but NOT the tmux server or broker port, so any change verified mid-session
gets a spurious failure on these E2E targets — a false negative that can
trigger a wasted fix-feedback cycle to an agent whose change is actually fine
(learnings-privacy only touches docs + a CLI notice). Fix options: verify-gate
runs E2E against an isolated tmux socket (`-L <uniq>`) + a free/ephemeral
broker port; or the gate detects a live paw session and runs E2E serially
after pausing it; or excludes live-colliding E2E targets with a logged caveat.
Relates to cold-start-ci-parity, [[feedback_serialize_e2e_verification_runs]],
[[feedback_local_pass_doesnt_equal_ci_pass]]. → **v0.8.0** (verification
isolation). WATCH-OUTCOME (resolved): the supervisor diagnosed it CORRECTLY on its own —
caught that the task-notification's "exit 0" was masked by a trailing echo
(cited its own background-pipe-masks-exit-code learning), noted baseline
`just verify` passed and LP only touches docs+notice, and concluded "likely an
E2E/live-session race, not a real regression — must find what actually failed
before judging." So no spurious LP fix-request. BUT the finding stands: the
supervisor burned cycles diagnosing a false-negative that env-isolation would
prevent entirely. Fix = isolate the verify tmux socket + broker port.

## F7 — Coding agents reflexively attempt /opsx:archive after committing
Both finished agents (learnings-privacy, then broker-live-watch) typed
`/opsx:archive <change>` once their work was committed. opsx-role-gating
(v0.6.0) gates *execution* (the skill refuses for a coding-agent role) and the
supervisor catches each attempt with a role-gating feedback nudge — but the
agent still wastes a turn composing it and the supervisor must intervene every
time. Refinement: the bundled coding-agent skill should explicitly say "after
your final commit, STAND BY — do NOT run /opsx:verify or /opsx:archive; the
supervisor verifies + archives," so the gate is hit proactively (agent never
attempts) rather than reactively (supervisor mops up). Pairs with the F6/drift-F
approval-channel work. → **v0.8.0** (skill wording; cheap, could also ride
earlier). Positive note: the gating + supervisor feedback DID hold both times —
no premature archive occurred.

## F6 — Approvals should be broker "chat-with-options", not per-pane CLI modals
User idea (2026-06-19): instead of the CLI's blocking multiple-choice
permission modal (which pulls the supervisor/operator out of the main window
into each agent pane), have the agent surface the decision as a broker/chat
message with options that the supervisor answers in-band. Key facts: the
modal is the CLI permission layer (not agent-controllable via prompt), and
there is no CLI feature to render an approval as a chat message/structured
output. So this is a **coordination-layer** design, not a CLI tweak — and
git-paw already has the primitives (`agent.question` / `agent.blocked` +
supervisor response over the broker). Realisation: agents publish a proposed
gated action to the broker; supervisor approves/denies in its own flow; agent
polls and proceeds. This IS drift F (broker-as-trigger / supervisor-as-actor)
and subsumes F2 (auto-approve race) + F5 (stray send-keys). → **v0.8.0**.

Stop-gaps available TODAY (no new code): (a) `agent_approval = "full-auto"`
in config → CLI launches with `--dangerously-skip-permissions`, zero modals
(acceptable for worktree-isolated + branch-guarded agents; trades away the
auto-approve dogfood signal); (b) broaden `permissions.allow` prefix rules +
`acceptEdits` permission mode so the dev loop never prompts but novel/risky
ops still do. The current dogfood runs `agent_approval = "auto"` on purpose to
exercise the approve subsystem — which is why modals appear.

## F4 — Dashboard agent-table "Summary" column is dead, remove it
The agent-status table's "Summary" column always renders blank in production:
the broker hardcodes `summary: String::new()` when building each `/status`
roster entry (`src/broker/delivery.rs:417`), and `StatusPayload` (the
`agent.status` message) carries no summary field at all. Remove: the "Summary"
header + cell (`src/dashboard.rs:327`, `:344`), the `summary` field on the row
struct (`:71`) and its copy (`:155`), and the hardcoded empty at
`delivery.rs:417`. **Do NOT touch** `broker_log.rs::derive_summary`, nor the
`IntentPayload.summary` / `AdvancedMain.summary` message fields — those are a
separate, used concern (broker-log digests). → small dashboard-polish change,
pairs with F3. Candidate slug: `dashboard-drop-summary-column`.

## F11 — MCP server has no docs/README access tool
The mcp-read-tools surface (coordination/governance/project-knowledge/session/git)
exposes specs/tasks/skills but NOT the repo's README or mdBook docs (docs/src/*.md).
For a standalone "developer-MCP-for-this-repo" (a client may have ONLY git paw mcp
connected, no filesystem MCP), an agent can't read the README or the MCP setup
guide it just wrote. Gap. Proposed additive tools in mcp-read-tools:
get_readme(), list_docs() (walk docs/src/SUMMARY.md), get_doc(path) — repo-root
confined like get_spec (reuse the path-traversal guards). → v0.7.x ride-along or v0.8.0.
Candidate slug: mcp-doc-tools.

## F12 — Governance is bring-your-own; git-paw ships no governance scaffolding
[governance] is a pure path-pointer (adr/test_strategy/security/dod/constitution,
all optional, default None) at the USER's own files. git paw init scaffolds NONE of
them (confirmed: init.rs has no governance scaffolding). git-paw the product defines
no ADRs/security-checklist/test-strategy; this repo only wires dod=AGENTS.md. So the
governance MCP/supervisor tools return empty unless a user authors + configures their
own docs. PRODUCT DECISION: keep BYO (flexible, unopinionated) vs. add `git paw init`
governance scaffolding (starter adr/, security-checklist.md, test-strategy.md, dod.md
templates + auto-wire [governance]) so the supervisor's spec-audit + MCP governance
tools have content out of the box — parallels how git-paw is opinionated about specs.
→ product-direction call; if yes, v0.8.0. Candidate slug: governance-scaffolding.

## F13 — MCP server advertises SDK-default identity "rmcp", and the name isn't configurable
`GitPawMcpServer::get_info()` builds `ServerInfo::default()` and overrides only
`capabilities` + `instructions`, never `server_info.name`/`version` — so the
initialize handshake advertises `serverInfo: { name: "rmcp", version: "1.7.0" }`
(the rmcp SDK default), not git-paw. Two parts:
- BUG (fix regardless): set server_info.name = "git-paw" and version =
  env!("CARGO_PKG_VERSION") in get_info(). The server currently misrepresents
  its identity in the protocol + logs.
- ENHANCEMENT (user request): make the advertised name configurable for
  multi-repo disambiguation. Options: a `--name <NAME>` flag on `git paw mcp`
  (fits the client-spawned model, lives beside --repo/--log-file), or a per-repo
  `[mcp] name` config field. Default "git-paw"; could auto-suffix the repo
  basename. NOTE: the *display* name in Claude Desktop is the mcpServers KEY in
  the client config (already user-controlled) — this is about the server's own
  advertised serverInfo.name.
→ fix the identity bug in v0.7.0 (part of the MCP headline; the server
shouldn't claim to be "rmcp"); configurable name can ride along or go v0.8.0.
Candidate slug: mcp-server-identity.

## F14 — MCP server can't browse source code (no file read / tree listing)
The read surface exposes specs, docs, coordination, governance, session, and
git diffs/log — but no way to LIST the source tree or READ an arbitrary source
file. A client connected only to git paw mcp cannot explore src/ (observed:
Claude Desktop fell back to cloning the public repo). Gap in the
"developer-MCP-for-this-repo" thesis — exploring the actual code is the most
common need. Proposed new tool category (Source/Files) in mcp-read-tools:
- list_files(subpath?) — `git ls-files --cached --others --exclude-standard`
  (tracked + untracked-not-ignored; gitignored target/secrets EXCLUDED),
  optionally scoped to a subpath.
- read_file(path) — read one file, repo-root-confined (canonicalize+starts_with
  guard, reuse get_doc's) AND gitignore-respecting (refuse ignored paths). Reads
  the LOCAL working tree → actual branch/uncommitted state.
- (optional) search_code(query) — grep across tracked files.
Security: exposes committed + working source on the user's own machine to a
client they spawned; gitignore + repo-confinement keep secrets/build out.
→ scope/placement: user request (2026-06-22). Candidate slug: mcp-source-tools.

### F10 — ROOT CAUSE CONFIRMED (2026-06-22): product behavior, not dogfood
Traced to src/agents.rs: during worktree setup git-paw injects its managed
`<!-- git-paw:start -->` skill section into the worktree's AGENTS.md, then (line
~281) calls `assume_unchanged(worktree_root, "AGENTS.md")` + adds it to
`.git/info/exclude` — deliberately, to stop `git add -A` from committing the
ephemeral injection. The guard is file-level, so it hides ALL changes to
AGENTS.md, blocking LEGITIMATE committed edits too. This is general product
behavior — it bites any project where an agent must commit an AGENTS.md edit
mid-session (git-paw's own repo just hits it most, since AGENTS.md is actively
maintained). NOT a dogfood artifact. Fix directions: (a) inject into a
gitignored sidecar instruction file rather than the tracked AGENTS.md, so the
tracked doc is never hidden; (b) on a real agent commit, detect+strip the
injected block and clear assume-unchanged for that commit; (c) document the
escape hatch (`git update-index --no-assume-unchanged AGENTS.md`). → real
product fix; needs a small design pass. v0.7.x or early v0.8.0.
