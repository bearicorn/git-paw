# v0.5.0 — archive order

This document records the safe dependency order for archiving the v0.5.0
changes during the `chore: prepare v0.5.0 release` commit. It exists so
the release engineer can run `openspec archive <change> -y` in an order
that does not leave specs referencing types or functions that have not
been merged into the main specs yet.

## Constraints

- `spec-kit-format` exports `git_paw::specs::speckit::detect_constitution`.
  `governance-config` imports it. **`spec-kit-format` MUST archive
  first.** Archiving `governance-config` first would leave a delta that
  references a function not present in the main specs.
- `governance-context` is the runtime consumer of `[governance]` paths
  stored by `governance-config`. **`governance-config` MUST archive
  before `governance-context`** so the storage slot exists in the main
  specs when the runtime delta lands.
- The remaining v0.5.0 changes (`cross-format-spec-selection`,
  `forward-coordination`, `learnings-mode`, `no-supervisor-flag`,
  `supervisor-as-pane`, `v040-hardening`, `conflict-detection`,
  `prompt-submit-fix`) do not interact with the governance/speckit
  dependency chain and can be archived in any order — group them at the
  end of the run for predictability.

## Safe sequence

Run the archive commands in this order during release prep:

```bash
# 1. SpecKit backend lands first — it exports detect_constitution.
openspec archive spec-kit-format -y

# 2. Governance storage slot — consumes detect_constitution.
openspec archive governance-config -y

# 3. Governance runtime consumer — consumes the [governance] paths.
openspec archive governance-context -y

# 4. Independent changes — order does not matter, but keep
#    deterministic by archiving alphabetically.
openspec archive conflict-detection -y
openspec archive cross-format-spec-selection -y
openspec archive forward-coordination -y
openspec archive learnings-mode -y
openspec archive no-supervisor-flag -y
openspec archive prompt-submit-fix -y
openspec archive supervisor-as-pane -y
openspec archive v040-hardening -y
```

If any archive step fails because a delta references a requirement that
does not exist in the target spec, fix the delta header
(`## ADDED Requirements` vs `## MODIFIED Requirements`) before
re-running the archive — do not reach for `--skip-specs` unless the
implementation is already in code and the spec content is purely
informational.

## Release notes copy

The `chore: prepare v0.5.0 release` commit body should call out the
governance changes explicitly, since they add a new TOML table that
project teams will discover in CHANGELOG.md:

> **New `[governance]` table.** Optional path pointers to your
> project's existing governance documents (ADRs, test strategy,
> security checklist, Definition of Done, constitution). All five
> fields default to `None`; v0.4 configs continue to load unchanged.
> git-paw does not generate, vendor, or template any of these
> documents — it stores the paths and the supervisor reads them as
> context during its audit.
>
> **Spec Kit constitution auto-wiring.** When `[specs] type =
> "speckit"` and `governance.constitution` is unset, git-paw probes
> for `<specs_dir>/../memory/constitution.md` and populates the field
> automatically. Explicit user values always win, including the empty
> string (which suppresses auto-wiring without deleting the slot).
>
> **Runtime usage ships in `governance-context`.** The boot-prompt
> injection and supervisor audit handling that make the configured
> docs actually do anything live in the parallel
> `governance-context` capability — `governance-config` ships the
> storage slot only.

These three paragraphs map to the three bullet points in the v0.5.0
CHANGELOG section produced by `git cliff`.
