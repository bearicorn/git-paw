# v0.4.0 OpenSpec Archive Order

This document records the dependency-respecting order in which the 24 active
changes under `openspec/changes/` should be archived for the v0.4.0 cut. It is
generated from a manual delta-graph analysis. Do **not** run `openspec archive`
based on intuition — follow the chains below.

A change A *precedes* change B when:
1. They both write deltas against the same capability, **and**
2. A modifies (or adds, then B modifies) a requirement whose final text B
   depends on, **or** B's MODIFIED replacement text supersedes A's wording in a
   way that requires A to be applied first.

Disjoint deltas (different requirements within the same capability, or
different capabilities entirely) are treated as **independent** and may be
archived in any order or in bulk.

## 4. Ordered chains

### Chain A — `agent-skills` capability

Five changes touch `agent-skills`. Two requirements drive ordering:

- `Embedded supervisor skill`: ADDED by **supervisor-skill**, MODIFIED by
  **spec-audit** (adds the "Spec Audit Procedure" section).
  → supervisor-skill MUST precede spec-audit.

- `Embedded coordination skill`: MODIFIED by **supervisor-agent**,
  **hook-injection**, **supervisor-messages**. Each rewrites the body of the
  same requirement. Last write wins, so the *last* archived change determines
  the final requirement text in the main spec.

Recommended chain:

```
supervisor-skill  →  supervisor-agent  →  hook-injection  →  supervisor-messages  →  spec-audit
```

Rationale:

1. **supervisor-skill** introduces the supervisor skill (must exist before
   spec-audit modifies it).
2. **supervisor-agent** lays the v0.4 baseline coordination skill (proactive
   status at three triggers, cherry-pick section).
3. **hook-injection** removes the manual "MUST publish agent.status" clause
   (now automated by the filesystem watcher) — this is a *narrowing*
   modification that depends on supervisor-agent's three-trigger text being
   the prior state.
4. **supervisor-messages** adds the "Messages you may receive" section
   documenting `agent.verified` and `agent.feedback`. Its MODIFIED text is
   purely additive and is safe to apply last.
5. **spec-audit** finally modifies the supervisor skill (separate requirement,
   `Embedded supervisor skill`, not `Embedded coordination skill`) so its
   placement at the end of the chain is non-conflicting.

> **Composition warning** flagged for human review: each MODIFIED on
> `Embedded coordination skill` replaces the requirement body. If the
> author of supervisor-messages did not include the proactive-trigger and
> auto-status-removal language from the prior two changes in its replacement
> text, the final main spec will *lose* that language after archive. Verify
> by diffing the three MODIFIED bodies before running `openspec archive`. If
> a merge is required, edit `supervisor-messages/specs/agent-skills/spec.md`
> to include the union before archiving.

### Chain B — `broker-messages` + `message-delivery` capabilities

Two changes share both: **prompt-inbox**, **supervisor-messages**.

- `Broker message envelope` is MODIFIED by prompt-inbox ("extended with a
  fourth variant `Question`"). supervisor-messages does *not* MODIFY the
  envelope; it ADDS sibling requirements `Verified message variant` and
  `Feedback message variant`. No textual conflict.
- `message-delivery`: prompt-inbox MODIFIES the three existing routing
  requirements + ADDS `Question messages are delivered to the supervisor
  inbox`. supervisor-messages ADDS three sibling requirements (`Verified
  ... broadcast`, `Feedback ... target`, `Agent record updated...`). No
  conflict.

Recommended order:

```
prompt-inbox  →  supervisor-messages
```

Rationale: prompt-inbox MODIFIES baseline requirements; supervisor-messages
only ADDS. Putting prompt-inbox first keeps the MODIFIED→ADDED progression
clean and matches the wording of prompt-inbox's envelope text ("fourth
variant Question"), which leaves room for the additional Verified/Feedback
variants supervisor-messages then adds.

### Chain C — `dashboard` capability

Two changes: **prompt-inbox**, **align-specs-with-v040-impl**. Both MODIFY
the `Dashboard entry point` requirement with conflicting `run_dashboard`
signatures:

- prompt-inbox: `pub fn run_dashboard(state: BrokerState, broker_handle: BrokerHandle) -> Result<...>` (no shutdown flag)
- align-specs-with-v040-impl: `pub fn run_dashboard(state: &Arc<BrokerState>, broker_handle: BrokerHandle, shutdown: &AtomicBool) -> Result<...>` (final v0.4 signature, matches `src/`)

`align-specs-with-v040-impl` MUST be archived **after** prompt-inbox so its
signature wins. Same goes for the `Periodic state polling` requirement —
align-specs has the final v0.4 wording.

Recommended order:

```
prompt-inbox  →  align-specs-with-v040-impl
```

### Chain D — `configuration` capability

Two changes: **supervisor-config**, **align-specs-with-v040-impl**. They
modify *different* requirements within `configuration`:

- supervisor-config MODIFIES `Parse TOML config with all fields` and
  `Default config generation`.
- align-specs-with-v040-impl MODIFIES `Specs configuration section`.

Disjoint requirements → no structural conflict. Either order would archive
cleanly. Recommended order:

```
supervisor-config  →  align-specs-with-v040-impl
```

Rationale: align-specs is the v0.4-final cleanup pass and should land last
across **all** capabilities it touches (broker-server, configuration,
dashboard, error-handling) so its final-state spec text wins everywhere.

## 7. D18 — Decision on `docs-skill-templates-prerender`

**Decision: DELETE** — the work it tracks is already merged.

### Evidence

- The directory contains only `tasks.md`. There is no `proposal.md`, no
  `specs/`, no delta — `openspec validate docs-skill-templates-prerender
  --strict` reports "Unknown item". It cannot be archived via
  `openspec archive`.
- Its `tasks.md` summary states the goal as "Update hook-injection specs to
  reflect common git dir usage" and "Update supervisor-mode tasks.md to
  match current signatures."
- Commit `67cb19b docs(specs): fix documentation drift for hook-injection
  and supervisor-mode` (already on `feat/v0.4.0-supervisor`) modifies
  exactly:
  - `openspec/changes/hook-injection/specs/git-hook-injection/spec.md`
  - `openspec/changes/supervisor-mode/tasks.md`
  This is the same scope listed under SHALL items 1 and 2 of
  docs-skill-templates-prerender.
- SHALL items 3 ("Ensure all skill templates match current implementation")
  and 4 ("Update user-facing documentation") are either covered by
  `standardize-agent-skills` (skill-standardization, skill-validation
  capabilities) or are general docs hygiene tasks better tracked outside
  OpenSpec.

### Why not "fold into another change"

The natural folding target would be `standardize-agent-skills` (skill
template consistency) or `hook-injection` (where the spec drift already
landed). However, `standardize-agent-skills` is already a clean, validated
change ready for archive — folding tasks.md content in would re-open it.
And the actual code/spec edits are already committed via 67cb19b. There is
no remaining work to track.

### Why not "archive as-is"

It cannot be archived as-is. It has no proposal.md and no spec deltas, so
`openspec archive` will refuse it. Promoting it to a real OpenSpec change
just to immediately archive it would document work that is already in
the git log.

### Action

Delete the directory after this branch ships:

```
rm -rf openspec/changes/docs-skill-templates-prerender
```

(Not deleted by this document — only documented. The deletion should land
in the same PR as the v0.4.0 archive sweep.)

The same reasoning likely applies to the seven other tasks-only folders
(`fix-*`, `test-*`). Each should be cross-checked against recent commits
before bulk deletion. That audit is out of scope for D17/D18 and is left
for a follow-up.
