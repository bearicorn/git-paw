## Why

The skill templates module (`src/skills.rs`) can load and render coordination instructions, but nothing calls it. The per-worktree `AGENTS.md` generated by `src/agents.rs` still only contains the v0.2.0 content (assignment, spec, file ownership). This change closes the last gap: when broker is enabled, the coordination skill is resolved, rendered with the worktree's branch name, and embedded inside the existing git-paw marker section so the coding CLI reads it automatically on startup.

## What Changes

- Extend `WorktreeAssignment` in `src/agents.rs` with an optional `skill_content: Option<String>` field containing the pre-rendered coordination skill text
- Extend `generate_worktree_section` to append the skill content inside the `<!-- git-paw:start/end -->` markers after the existing assignment, spec, and file ownership sections
- In the session launch flow (`src/main.rs`), when `[broker] enabled = true`:
  - Call `skills::resolve("coordination")` to get the template
  - For each worktree, call `skills::render(&template, &branch, &broker_url)` to produce the rendered skill text
  - Pass the rendered text into `WorktreeAssignment.skill_content`
- When broker is disabled, `skill_content` remains `None` and the generated section is identical to v0.2.0

## Impact

- **Modified file:** `src/agents.rs` — extend `WorktreeAssignment` struct and `generate_worktree_section` function
- **Modified file:** `src/main.rs` — call `skills::resolve` and `skills::render` in the launch flow, pass result into `WorktreeAssignment`
- **No new files, no new modules, no new dependencies.**
- **Depends on:** `skill-templates` (for `skills::resolve`, `skills::render`, `SkillTemplate`)
- **Merge order:** can merge in any order relative to `peer-messaging` and `broker-integration` in Wave 2, since it only touches `src/agents.rs` and the launch flow. However, it should merge before `broker-integration` so that when `broker-integration` wires up the full session, skills are already being injected.
- **Backward compatible:** `skill_content: None` produces the same output as v0.2.0. Existing tests pass unchanged.
