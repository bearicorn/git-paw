## Context

The v0.4.0 Wave 1 changes (`supervisor-config`, `supervisor-skill`, `supervisor-messages`) defined config structs, the supervisor AGENTS.md template, and new message types. This change is the runtime implementation: it wires those pieces together to actually launch the session. Without this change, supervisor mode is purely configuration with no execution.

The key insight from v0.3.0 dogfooding: agents fail at coordination when they're reactive (only publishing status when asked) and when they have no rules about file ownership. This change bakes both behaviors directly into every agent's AGENTS.md.

## Goals / Non-Goals

**Goals:**

- Implement `cmd_supervisor()` — the full auto-start flow
- Inject `GIT_PAW_BROKER_URL` into each tmux pane's environment
- Construct CLI launch commands with approval flags from config
- Add inter-agent rules to the generated `AGENTS.md` coordination section
- Update `coordination.md` with proactive status publishing and cherry-pick instructions

**Non-Goals:**

- Supervisor verification/feedback logic (belongs in the supervisor skill at runtime)
- Merge ordering (owned by `supervisor-mode`)
- Session summary generation (owned by `session-summary`)
- Interactive supervisor mode resolution (owned by `supervisor-mode`)

## Decisions

### Decision 1: Supervisor CLI runs in the foreground terminal, agents run in detached tmux

The supervisor CLI is the operator's interface — it receives the final merged view and can ask questions. It must be in the user's interactive terminal. Agents are background workers; they run in tmux panes and are observed via the dashboard (pane 0).

**Implementation:**
1. Create tmux session in detached mode
2. Wait ~2s for panes to boot
3. Inject initial prompts via `tmux send-keys`
4. Start supervisor CLI in the foreground (blocking call)

**Alternatives considered:**
- *Supervisor in pane 0 instead of dashboard.* Loses the status overview. Rejected.
- *All panes detached, operator attaches manually.* No foreground control. Rejected.

### Decision 2: Inject `GIT_PAW_BROKER_URL` via `tmux set-environment` before pane launch

Each coding agent pane needs the broker URL to publish status. Using `tmux set-environment -t <session>` before creating panes propagates the variable to all panes without requiring each launch command to carry it explicitly.

**Why:**
- Shell-standard environment propagation — no changes to CLI launch commands needed
- The coordination skill template already uses `{{GIT_PAW_BROKER_URL}}` as a placeholder (substituted at render time), so the variable must be present in the shell environment at run time

**Alternatives considered:**
- *Hardcode URL in each `tmux send-keys` command.* Fragile — if the broker port changes, all injected commands need updating. Rejected.
- *Write URL to a file agents can read.* Extra I/O and agents need to know the file path. Rejected.

### Decision 3: Inter-agent rules are injected as a dedicated section in the generated AGENTS.md

The coordination skill already covers how to call the broker API. Inter-agent rules (file ownership, never push, proactive status, cherry-pick) are behavioral constraints that belong in a separate `## Inter-Agent Rules` section generated by `src/agents.rs`.

**Why:**
- Keeps the coordination skill template generic and reusable outside supervisor mode
- The rules section is always present when generated by the supervisor; regular `git paw start` can omit it
- Clear separation of "how to communicate" (coordination skill) vs. "rules of engagement" (inter-agent rules)

**Alternatives considered:**
- *Embed rules in the coordination skill template.* Contaminates the skill with supervisor-specific policy. Rejected.
- *Runtime rules from the broker.* Adds complexity without benefit — rules are static per session. Rejected.

### Decision 4: Proactive status publishing uses three trigger points

The `coordination.md` template is updated to require `agent.status` at three points:
1. When starting work on a new file or task
2. After editing or creating any file (include `modified_files`)
3. After each `git commit`

This matches the data the supervisor and dashboard need without flooding the broker with unnecessary messages.

**Alternatives considered:**
- *Publish on every file read.* Too noisy, degrades broker performance. Rejected.
- *Only on commit.* Too infrequent — peers can't detect conflicts until too late. Rejected.

### Decision 5: Cherry-pick is the inter-agent dependency resolution mechanism

When agent A needs work from agent B, A publishes `agent.blocked`. When B's artifact arrives in A's inbox, A must cherry-pick B's commit rather than waiting for the supervisor to merge. This keeps agents unblocked without requiring the supervisor to intervene for every dependency.

The coordination skill template is updated with the exact `git cherry-pick` command agents should run.

**Alternatives considered:**
- *Wait for supervisor merge.* Serializes work unnecessarily. Rejected.
- *Agents push to shared branch.* Conflicts with "never push" rule. Rejected.

## Risks / Trade-offs

- **2-second boot wait is a magic constant** — if the system is slow, panes may not be ready. Mitigation: the initial prompt injection retries on `tmux send-keys` failure; the 2s is a floor, not a hard timeout.
- **Injecting prompts via `tmux send-keys` is inherently racy** — the agent CLI must be in input mode. Mitigation: the launch command starts the CLI, which prompts for input before the user types anything; the 2s delay is usually sufficient.
- **`GIT_PAW_BROKER_URL` must be set before pane creation** — if broker startup is slow, the URL may not be known yet. Mitigation: broker starts first and returns its URL before tmux session creation begins.
